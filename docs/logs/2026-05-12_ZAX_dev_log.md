## 1. サーバーエラー `MIDDLEWARE_INVOCATION_FAILED` の修正

### 症状
Vercel本番環境において、ページにアクセスした際に以下の500エラー（内部サーバーエラー）が発生しました。
`tk22z-1778522712151-0812915b7168500 : 内部サーバーエラーコード：MIDDLEWARE_INVOCATION_FAILED`

### 原因
- Next.js の `middleware.ts`（Vercelの Edge Runtime で動作する仕組み）において、Node.js専用モジュールである `crypto` を利用するセッション署名関数（`signSession`）をインポートし、実行していたため。
- Edge Runtime はブラウザに近い軽量な実行環境（V8ベース）であるため、Node.jsのネイティブモジュールである `crypto`（AES-256やHMAC等の重い暗号関数）に非対応であり、実行時エラーを引き起こしていました。

### 対策
- 開発・テスト段階における動作安定化を最優先とするため、`middleware.ts` から `crypto` 依存の暗号化署名処理を排除。
- セッション管理はシンプルな UUID（標準の `uuid` ライブラリはEdge互換性あり）のみで行う設計へ変更。
- **ファイル変更**: `src/middleware.ts`

## 2. AIおみくじ結果からの Markdown 記号（`#`、`*`、`**`）の完全な排除

### 症状
Gemini APIが返却する分析レポート・おみくじの中に、マークダウンのヘッダー記号（`#`）や太字指定（`**`）、リスト記号（`*`）などの不要な装飾記号が混入し、UIを損ねていました。

### 原因
- プロンプトレベルの出力指定のみでは、LLM（Gemini）が時折Markdown記号を出力してしまう問題。
- API側での置換置換処理（`.replace(/[*#]/g, '')`）が十分ではなく、二重の太字（`**`）や複数シャープ（`###`）を完璧にカバーしきれていませんでした。

### 対策
- レポート生成APIのプロンプト内に「JSON形式のテキスト部分には角括弧、シャープ、アスタリスクなどのMarkdown記号を一切使用するな」とさらに厳密な命令を追加。
- 出力されたテキストに対し、より確実かつアグレッシブにMarkdown記号を除去するための正規表現置換を追加。
  ```typescript
  rawText = rawText
    .replace(/```json\s*/gi, '')
    .replace(/```\s*/g, '')
    .replace(/\*\*/g, '')
    .replace(/\*/g, '')
    .replace(/#{1,6}\s/g, '')
    .replace(/#/g, '')
    .replace(/_{1,2}/g, '')
    .trim();
  ```
- **ファイル変更**: `src/app/api/diagnostic/generate-report/route.ts`

## 3. 開発フェーズに向けた診断質問数の簡略化（6問化）

### 目的
診断の動作確認、マッチング確認、おみくじレポート検証を高速に行うため、質問回答フローのステップ数を大幅に短縮。

### 対策
- 質問データファイル `src/data/questions.ts` を編集。
- 従来の全60問のデータは `questions_full` 配列として安全にファイル内に退避・保存。
- メインの診断に使用される `questions` 配列を、6つの診断カテゴリ（生活基盤、社会意識、信頼構築、対話力、野心、寛容性）から1問ずつ厳選した**計6問**にシンプル化。
- これにより、診断がわずか6ステップで完了し、レポート生成や価値観マッチングの検証をすぐにテストできるようになりました。
- **ファイル変更**: `src/data/questions.ts`
