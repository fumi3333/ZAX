## 1. サーバーエラー `MIDDLEWARE_INVOCATION_FAILED` の修正

### 症状
Vercel環境において、ページにアクセスした際に500内部サーバーエラーが発生。

### 原因
`middleware.ts` が Edge Runtime 環境で動作する際、Node.js専用のネイティブモジュールである `crypto` を利用する関数をインポート・実行していたため。

### 対策
ミドルウェアから `crypto` 依存の処理を完全に除外し、Edge環境と完全な互換性を持つ UUID によるシンプルなセッション管理へ移行しました。
*   **ファイル変更**: `src/middleware.ts`

## 2. クライアント側での React 例外（例外エラーによる白画面）の完全防止策

### 症状
診断を送信した後、結果画面に遷移する直前、または表示待機中に「クライアント側で例外が発生しました（zax.fumiproject.dev）」のNext.jsエラー画面が表示されていました。

### 原因
1.  **Rechartsのハイドレーションエラー**:
    `ResponsiveContainer` や `RadarChart` が、サーバーサイドでのHTML生成とクライアントサイドでのマウントで差分を生み、Reactのハイドレーションミスマッチ・クライアントクラッシュを引き起こしていた。
2.  **非同期データへの不完全なアクセス**:
    結果データ（`data`）のフェッチ完了前、あるいは失敗して fallback セッションストレージを読み込む際のプロパティ処理に `try-catch` やオプショナルチェイニングが一部不足しており、JavaScriptのエラーを誘発していた。

### 対策
1.  **マウント状態管理の導入**:
    [`ResultRadarChart.tsx`](file:///c:/ZAX/src/components/diagnostic/ResultRadarChart.tsx) に `mounted` フラグの状態管理（`useEffect`）を追加。サーバーサイドでの描画とマウント直前の不一致を完全に防ぎ、安全にチャートがマウントされるように堅牢化しました。
2.  **防衛的プログラムの徹底**:
    [`DiagnosticResultClient.tsx`](file:///c:/ZAX/src/components/diagnostic/DiagnosticResultClient.tsx) にて、APIから値を取得した際、またはパースの際のすべてのアクセスにオプショナルチェイニング (`?.`) と `try-catch` を適用。エラー時は真っ白な例外クラッシュ画面を出すのではなく、安全なローディング状態またはリトライボタンが動的に表示されるように修正しました。

## 4. 特殊記号（`*`、`#`、括弧、コロンなど）の徹底的な排除

### 症状
Gemini APIがおみくじテキストの内部に `*`、`#`、`()`、`[]`、`:` などの装飾用記号を出力し、デザインの安っぽさや視覚的なノイズを生んでいました。

### 対策
API側のプロンプト指示および置換処理に加え、フロントエンドのレンダリング直前の2重フィルターで完全排除しました。
1.  **API側 (`generate-report/route.ts`)**:
    プロンプトでの指示を強化。さらにJSON返却段階で、全てのテキストプロパティ（`otsuge`, `machihito`, `koudou`）から正規表現 `/[*#\[\]【】()（）:：]/g` を使って該当記号を除去。
2.  **フロントエンド側 (`DiagnosticResultClient.tsx`)**:
    AIから受け取った文字列を最終表示する直前に、`sanitizeText()` 関数を適用。記号を強制的に除去し、クリーンな自然文の日本語のみを表示するようにバリアを張りました。
*   **ファイル変更**: [`generate-report/route.ts`](file:///c:/ZAX/src/app/api/diagnostic/generate-report/route.ts), [`DiagnosticResultClient.tsx`](file:///c:/ZAX/src/components/diagnostic/DiagnosticResultClient.tsx)
