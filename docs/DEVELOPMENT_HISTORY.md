## 🤔 設計判断の理由

### 判断1: なぜ「補完性スコア」なのか

**背景**:
- 単純な類似度マッチングは「似た者同士」を引き合わせるだけ
- 心理学的に、似すぎた関係は成長を妨げる（確証バイアス）

**数学的根拠**:
- ベルカーブ（ガウス分布）で「適度な差異」を評価
- 最適類似度0.5は仮説（実データで検証予定）

**代替案との比較**:
- ユークリッド距離: 次元のスケールに依存（不適）
- マンハッタン距離: 外れ値に弱い
- コサイン類似度 + ベルカーブ: 方向性を評価、スケール不変

### 判断2: なぜ6次元なのか

**背景**:
- Big 5 (心理学の5因子モデル) を参考
- しかし、5次元では「柔軟性」が表現しづらい
- 6次元に拡張（Flexibility 追加）

**代替案**:
- 3次元: 単純すぎる
- 16タイプ（MBTI）: カテゴリカル（ベクトル化困難）
- 768次元（Embedding）: 人間が理解できない

**結論**: 6次元が「人間の理解可能性」と「表現力」のバランス

### 判断3: なぜPostgreSQL + pgvectorか

**代替案との比較**:

| 選択肢 | 利点 | 欠点 | 結論 |
|--------|------|------|------|
| **Pinecone** | 速い、スケーラブル | 月$70〜、リレーショナルなし | ×（コスト） |
| **Weaviate** | 高機能 | 別サーバー、学習コスト | ×（複雑） |
| **Chroma** | 軽量 | 本番環境では不安定 | ×（信頼性） |
| **pgvector** | PostgreSQL統合、無料 | 学習必要 | ✅ **採用** |

**決定理由**:
1. リレーショナルデータとベクトルを1つのDBで管理
2. トランザクション整合性
3. コスト削減（外部サービス不要）
4. 学習教材が豊富

### 判断4: なぜRAGで「継続学習」を代替するか

**本来の継続学習（不可能）**:
```
個人専用のGeminiモデルをファインチューニング
→ コスト: 数百万円/月
→ データ: 1人あたり数万サンプル必要
→ GPU: A100 x 8台
```

**RAGによる代替（可能）**:
```
過去の履歴をベクトル検索 → プロンプトに追加
→ コスト: ほぼゼロ
→ データ: 1回の体験から改善可能
→ GPU: 不要
```

**結論**: RAGで十分な効果が得られる

## 🐛 遭遇した問題と解決

### 問題1: PowerShellで `&&` が使えない

**症状**:
```powershell
npm run build && npm run start
# → エラー
```

**原因**: PowerShellは `&&` をサポートしていない

**解決**:
```powershell
npm run build; npm run start  # セミコロン使用
```

### 問題2: Prisma Client が見つからない

**症状**:
```
Prisma Client NOT FOUND
```

**原因**: `npx prisma generate` 未実行

**解決**:
```bash
npx prisma generate
```

### 問題3: ベクトル次元の不整合

**症状**:
- 診断: 768次元生成
- マッチングAPI: 6次元期待
- エラー発生

**解決**:
- 診断結果ページで6次元に変換
- カテゴリ平均を0-100にスケール変換

```typescript
const userVector6d = categoryScores.map(score =>
  Math.round(((score - 1) / 6) * 100)
);
```

### 問題4: Docker未起動時のビルドエラー

**症状**:
```
Database connection failed
```

**解決**:
- モックモードの実装（db/client.ts）
- DB未接続時は50人のアーキタイプを使用
- Vercel（本番）でも動作可能に

## 💡 開発中の気づき

### 気づき1: UI/UXの重要性

**当初**: 技術優先、UIは後回し  
**結果**: 見た目が悪いとデモで伝わらない

**学び**: ハッカソンでは「見せ方」が技術と同じくらい重要

### 気づき2: モックデータの価値

**当初**: 実DBでの動作を最優先  
**結果**: Docker起動忘れでデモ不可のリスク

**学び**: **デモはモックで動くべき**。本番DBは「あればベター」

### 気づき3: スコープの絞り込み

**当初の構想**:
- RAG実装
- TimescaleDB活用
- RLHF
- 音声分析
- BMI統合

**現実**:
- Phase 1: 診断 + マッチング + フィードバックフォーム
- 他は「将来展望」として残す

**学び**: **MVPは小さく、ビジョンは大きく**
